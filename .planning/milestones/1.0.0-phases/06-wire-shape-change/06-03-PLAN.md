<objective>
Rewrite `viewmodel-shell/src/browser.ts` as a thin interpreter of the new wire contract. Inputs read their value from local state at the declared `bind` path; user input events write back into local state at that path. Dispatch sends only the action name (plus state, plus files when present). The seven context-assembly sites identified by the codebase audit collapse into one declarative read/write seam.

Purpose: This is the renderer half of the wire-format break. Plan 01 changed the types; this plan changes the behavior. After this lands, an agent reading the JSON `{vm, state}` and walking the tree can mutate state and dispatch action names identically to how the browser does.
Output: A `browser.ts` with no `context`-assembly code paths, ~30-40% smaller; a shell-side `stateRead`/`stateWrite` seam over a dotted-path JSON-walk; the dispatch FormData no longer includes a `context` field in the `_action` JSON; jsdom tests exercise bind round-trips.
</objective>

This is the big one. The current `browser.ts` (961 lines) has seven distinct `context: { ... }` assembly sites at lines 321-352 (form harvest), 452 (select-on-change), 526 (field-on-Enter), 557 (checkbox-on-change), 634 (tab click), 762 (sort), 790 (filter), 882-888 (selection bulk action), 916 (pagination). All of these go away.

**Depends on Task 2:** the `stateAccess` parameter has been added to `Adapter.render` already. BrowserAdapter receives it as the third arg; store it on `this` (e.g. `this.stateAccess = stateAccess` at the top of `render`) and use it everywhere a bind value is read or written.

**The new pattern** (apply consistently to every input + dispatch node):

For an INPUT node (FieldNode, CheckboxNode, file input):
- On render: read the value from `this.stateAccess.read(node.bind)`. For a text input, set `input.value = String(read ?? "")`. For a checkbox, set `input.checked = Boolean(read)`. For a select, set `select.value = String(read ?? "")`.
- On user input event (`input`/`change`/`keyup`): write the new value back via `this.stateAccess.write(node.bind, newValue)`. For text: `newValue = input.value` (string). For checkbox: `newValue = input.checked` (boolean). For select-multiple: `newValue = Array.from(sel.selectedOptions).map(o => o.value)` (array of strings).
- If the node has an `action` field (FieldNode on Enter, CheckboxNode on change, select on change), ALSO dispatch the action name. The dispatch carries no extra data ŌĆö state already has the new value.

For a DISPATCH-ONLY node (ButtonNode, TabsNode tab, TableNode header/pagination/per-row):
- On click: `on({ name: <action-name> })` (or `on({ name, files })` when a file map is being attached on form submit). No context.

**Specific rewrites ŌĆö the 7 sites:**

1. **Form harvest (lines 321-352):** Delete the entire `harvest()` function. FormNode submit and FormNode.buttons[] entries become plain dispatches. The form's child inputs are bound to state via their individual bind paths; their values are ALREADY in state by the time the user clicks submit (because every keystroke wrote to state). On submit click: just `on({ name: submitAction.name })`. File inputs are the only thing that needs harvesting (see #2).

2. **File inputs (line 345-347):** Files don't fit in JSON state. The renderer still attaches files to the dispatched action's `files` map. New shape: on submit, walk the form's `input[type=file]` elements, build a `files: Record<string, File>` map from `name ŌåÆ input.files[0]`, and pass that as `action.files`. The dispatch path in `index.ts` (lines 475-479) appends them as multipart entries. Per the CONTEXT.md decision, when a file is picked, the file input's change handler ALSO writes `{ filename: file.name, size: file.size }` to state at the bind path (so the picked file is visible in state); the actual binary travels in `action.files`.

3. **Select-on-change (line 452):** When a select fires its FieldNode.action, just dispatch `{ name: action.name }`. The new selected value is already in state at the field's bind path (the change handler wrote it before dispatching).

4. **Field-on-Enter (line 526):** Same ŌĆö `{ name: action.name }`. Text value is in state.

5. **Standalone CheckboxNode (line 557):** Same ŌĆö `{ name: action.name }`. Checked state is in state at the checkbox's bind path.

6. **Tab click (line 634):** Each tab in TabsNode.tabs[] has its own action (Plan 01 reshaped this). On click, write `tab.value` to state at the TabsNode's bind path, then dispatch `{ name: tab.action.name }`.

7. **Table sort/filter/pagination (lines 762, 790, 916):**
   - **Sort:** on header click, write `{column: col.key, direction: nextDir}` to state at `n.sortBind`, then dispatch `{ name: n.sortActions[col.key].name }`. Plan 01 left `sortBind`, `sortActions` on TableNode for exactly this.
   - **Filter:** Plan 01 kept `TableNode.filterAction?: ActionEvent` (one filter dispatch action per table). On Enter in any column's filter input, the per-column filter value is already in state at `n.filterBinds[col.key]` (the input's `input` handler wrote it on keystroke); dispatch `{ name: n.filterAction.name }` ŌĆö no per-column action plumbing.
   - **Pagination:** prev/next clicks: write target page number to `<n.paginationBind>` and dispatch `{ name: n.pagination.prevAction.name }` or `nextAction.name`. (Plan 01 added prevAction/nextAction on TablePagination and paginationBind on TableNode.)

8. **Bulk selection action (lines 882-888):** This whole code path is gone. `TableSelection` was removed in Plan 01. Per-row checkboxes are rendered via the existing CheckboxNode path with their `bind` pointing at the app's selection slot; bulk "Delete Selected" buttons are plain ButtonNodes the app puts wherever it wants.

**Draft / focus / scroll preservation:**
- The existing focus/caret/scroll preservation logic (around the existing `restoreFocus` / scroll snapshot code ŌĆö look at lines 52-66 and elsewhere) STAYS ŌĆö it operates on the DOM, not on state, and continues to work.
- **Draft preservation:** the explicit `draftValues` map / snapshot/restore mechanism in current browser.ts is **deleted**. Drafts ARE state now ŌĆö on every keystroke, the input's value is written to state at its bind path; on re-render, the input reads from state. The result is equivalent draft-survival behavior for the case where the server response carries the in-place-mutated state back unchanged (the typical case). When the server explicitly OVERWRITES the bound state value, the input shows the server's value ŌĆö same as the old explicit-server-set-value path. The Task 4 jsdom test #12 verifies this.

**Net result:** The file should drop from ~961 lines to ~600 or so. Remove all helper functions related to context assembly. The render dispatch is now one consistent pattern: bind-driven read, event-driven write, action-only dispatch.

Walk every input/dispatch node type and apply the new pattern. Edge cases:
- Empty bind / missing bind: should not happen after Plan 01 ŌĆö every input node has a required `bind` field. If a node is missing a bind, that's a bug to fix at the type / demo layer; don't silently no-op.
- `inputType: "code"` ŌĆö same as text, just a different CSS class.
- `select-multiple` ŌĆö array of strings in state.
- `inputType: "hidden"` ŌĆö same pattern; reads from state, writes nothing (no user input).

Be diligent ŌĆö this is the biggest behavioral change in the milestone. Read the existing browser.ts thoroughly before editing.
