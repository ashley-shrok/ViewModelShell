The parity runner constructs multipart submissions from fixture step `action` objects. Currently each step looks like:
```json
{ "method": "POST", "action": { "name": "filter", "context": { "value": "active" } } }
```
Under the new shape, steps will be:
```json
{ "method": "POST", "action": { "name": "select-tab-active" }, "stateMutations": [{ "path": "filter", "value": "active" }] }
```

Two changes to the runner:

1. The action JSON appended to multipart's `_action` field should be `{name}` only — no context. Find the relevant code in `parity/run.ts` and update.

2. **New: fixture pre-dispatch state mutations (narrow scope).** Add a `stateMutations?: Array<{path: string, value: unknown}>` field to the step schema in `run.ts` — an optional array of pre-dispatch state writes. Apply each mutation to the prior step's response state (via the same dotted-path JSON walker the framework uses) before sending the step's request.

   **Scope this narrowly.** This mechanism exists **only for fixture steps that simulate user-typed form input before a submit click** — e.g., "user typed 'Acme' into the title field, then clicked Save" needs to inject the title value into state before dispatching the `add` action. Per-row actions, tab switches, sort/filter intents, and pagination clicks all use the server-derives-from-action-name pattern instead and need no state injection. Expect `stateMutations` to be used on roughly 10-20% of fixture steps (form submits) and absent from the rest.

   Implementation: walk the prior step's response state, apply each mutation via the same dotted-path write as the framework uses (re-use the helper from Plan 03 if exported; otherwise inline a small one in `run.ts` — 10 lines).

3. The response normalizer (`normalize.ts`) — probably no change needed; it normalizes the response shape (vm + state), and the response shape isn't changing in Phase 6. Confirm by reading the file.

4. The README at `parity/README.md` (if any) — update example fixture-step JSON to show the new shape.

For each fixture file, translate every step's `action` from the old shape (`{name, context}`) to the new shape (`{name}` plus optional `stateMutations`).

**Tasks fixture (parity/fixtures/tasks.json):**
- `filter-active`: was `{name: "filter", context: {value: "active"}}`. New: `{name: "select-tab-active"}` with `stateMutations: [{path: "filter", value: "active"}]`. (The mutation is technically redundant because the TabsNode renderer's bind would write state.filter — but the runner doesn't render; it just sends actions. So the mutation is required to keep the server's state model in sync.)
- `toggle-2`, `toggle-3`: were `{name: "toggle", context: {id: "2", checked: true}}`. New: `{name: "toggle-row-2"}` with `stateMutations: [{path: "items.<index-of-2>.completed", value: true}]`. The fixture needs to know the array index for the id — look at the demo's seed data; row id "2" is at items[1]. This is brittle if the seed changes; consider whether to fold the toggle action's semantics into the server-side handler so the mutation isn't needed in the fixture (the server can derive the right item by ID from the action name's suffix). Easier: have the server's "toggle-row-${id}" action read the id from the action name suffix and flip `items[match].completed` server-side; then the fixture step needs no mutation. Apply this approach.
- `delete-1`: was `{name: "delete", context: {id: "1"}}`. New: `{name: "delete-row-1"}` no mutation (server reads id from action name).
- Each tab-switching step: `{name: "select-tab-XXX"}` with state.filter mutation OR have the server's tab handler derive the filter from the action name suffix. Apply the latter for consistency with toggle/delete.

**Decision rule for fixtures: prefer server-derives-from-action-name over fixture-mutates-state.** This keeps fixtures cleaner and tests the action-name-encoding pattern end-to-end. Only use `stateMutations` for cases that truly need pre-set input values (form submits where the input was typed before the click).

**Apply this rule to all 7 fixtures.** Re-read each demo's action handler (from Plan 04) to know what the server expects.

**FeatureProbe** is the largest fixture — it exercises every feature. Expect to use `stateMutations` for form submit tests (where the form's input values would have been typed by a user — the fixture needs to inject them into state before the submit step).

**HelpDesk** is the canonical workflow fixture — tabs, filter narrowing, bulk action. Bulk-action steps need `stateMutations` to set `state.selectedIds` to a known set before dispatching `bulk-close`.

Be careful with the `${newId.<step-id>}` substitution syntax (line 5 of tasks.json) — the runner extracts new IDs from response state for subsequent steps. This still works under the new shape, just on state paths now instead of context payloads.

**Expected initial state:** likely several failures, because:
- A demo migration in Plan 04 may have left a subtle state-shape difference between .NET and bun.
- A fixture step may need a `stateMutations` entry that's missing.
- A demo's action-name encoding may differ between .NET and bun (e.g. one uses `delete-row-` and the other uses `delete/row/`).

**Triage workflow:** for each failure, read the diff. If two backends emit different normalized responses for the same step:
- Check the GET initial response — different state shape between backends? → Fix the demo with the wrong shape.
- Check post-dispatch response — different state-shape evolution? → Fix the handler with the wrong evolution.
- If both backends agree but the fixture expected something different → Fix the fixture or fold the expectation into a `stateMutations` entry.

Iterate until all 7 fixtures pass byte-identically across every backend.

Run the full `npm run check:core-globals` guard once at the end to make sure the renderer rewrite (Plan 03) didn't reintroduce a platform-global reference into core.
