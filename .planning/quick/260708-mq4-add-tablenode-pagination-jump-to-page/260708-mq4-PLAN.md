<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser client → server action dispatch | The jump input's typed value is untrusted client input; it rides the exact same `paginationBind` state path + action-name mechanism already used by prevAction/nextAction — no new trust boundary is introduced. |
| Direct wire caller (curl/agent) → server action dispatch | An agent can write ANY integer (including negative, zero, or huge) to `state.tablePage`-equivalent and POST `table-page-jump` directly, bypassing the browser's client-side clamp entirely. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-mq4-01 | Tampering | Direct-wire jump (agent bypasses browser clamp) | mitigate | Server-side clamp already exists and is unchanged by this plan (`Math.Clamp`/`Math.min+Math.max` in `Window()`/`tableWindow()`); Task 2's parity fixture step `tbl-page-jump-clamp` proves an out-of-range page (99) is still clamped to a valid page server-side in BOTH backends — the client-side clamp in Task 1 is a UX nicety, not the security boundary. |
| T-mq4-02 | Denial of Service | Malformed/non-numeric jump input | accept | The browser-side handler ignores non-numeric/empty input (no dispatch); a direct-wire caller sending a non-integer value to the bind path is an app-level state-shape concern identical to the pre-existing prevAction/nextAction risk (any app already trusts `TablePage`/`tablePage` to be an int from its own `Window()`/`tableWindow()` bind-write contract) — no new exposure introduced by adding a third action name to the same bind path. |
</threat_model>
