<objective>
Add the `blocking` field to `ActionEvent` and rework the client-side dispatch loop in `viewmodel-shell/src/index.ts` so a non-blocking (silent) round trip coexists with a blocking (user) dispatch instead of contending for a single shared mutex — today's `if (this.dispatching) return;` silently drops whichever dispatch arrives second, blocking or silent, and this plan replaces it with two independent in-flight lanes. Rapid non-blocking triggers coalesce to at most one extra in-flight request ("latest wins"), and a client-side monotonic sequence counter discards a stale/out-of-order response instead of letting it clobber a newer already-applied render. `browser.ts` is then updated so every trigger (checkbox, button, field, tabs, section, table) forwards the full `ActionEvent` object — including `blocking` — to `dispatch()`, since several existing call sites currently reconstruct a bare `{name: ...}` literal and would otherwise silently strip the new field.

This is the TS-only "core mechanism" half of Phase 14. The .NET wire twin (`ActionDescriptor.Blocking`) is delivered independently in Plan 14-02 (no shared files, same wave); the cross-backend parity fixture proving the field's wire shape is byte-identical lands in Plan 14-03 (wave 2, depends on both).

Design of record: `.planning/design/non-blocking-actions.md` (Stage 1 only — no wire epoch, no server-side reconciliation, no admission barrier; those are explicitly out of scope this phase). `pollInterval` itself is NOT folded into this path this phase (that is Phase 15, NBA-05) — but because `schedulePoll`'s existing `dispatch({name:"poll"}, true)` call already uses the `silent` parameter, it automatically routes through the SAME non-blocking lane built here, which is a documented, welcome side effect (not a scope addition — no code in `schedulePoll`/poll handling is touched).

Purpose: Deliver NBA-01 (coexistence — replace the single mutex), NBA-02 (coalescing/latest-wins), NBA-03 (client-side epoch discards stale/out-of-order responses), and the TS half of NBA-04 (the `blocking` field is a plain optional TS property — omitted when unset, exactly like every other optional wire field in this codebase; JSON.stringify already drops `undefined` keys, so "absent-when-default" requires no extra code on the TS side).

Output: `ActionEvent.blocking?: boolean` in `index.ts`; a two-lane (`blockingInFlight` / `nonBlockingInFlight`) dispatch loop with coalescing (`pendingNonBlockingRefire`) and epoch (`dispatchSeq` / `appliedSeq`) reconciliation, replacing the single `dispatching` mutex; every `browser.ts` trigger forwarding the full `ActionEvent`; two new vitest files proving NBA-01/02/03 at the dispatch-loop level and proving `blocking` propagates from every trigger node type. Full `npx vitest run` stays green — this is a behavior-preserving refactor for every app that never sets `blocking`.

CLOSED DESIGN OPEN-ITEM (`debounceMs`): the design of record lists "whether `debounceMs` is per-node" as an item to pin in planning. Decision for Phase 14: **do NOT implement a `debounceMs` field or any time-based debounce.** The coalesce-on-in-flight mechanism (`pendingNonBlockingRefire` — at most one extra round trip once the in-flight one resolves, carrying the latest trigger) fully satisfies NBA-02's "coalesce to a single in-flight request, latest wins" with zero timers and zero new wire surface. A time-based debounce is deferred as unneeded unless profiling later shows the coalesce-on-in-flight refire rate is a real problem; it would be a purely additive follow-up.
</objective>
