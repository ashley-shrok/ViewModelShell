This is the exact wire shape. Additive; protocol token stays `viewmodel-shell/1.0`.

--- TypeScript (viewmodel-shell/src/index.ts) ---

  export interface ChartPoint {
    label: string;   // category label (x-axis tick)
    value: number;   // numeric magnitude (bar height)
  }

  export interface ChartNode {
    type: "chart";
    /** Chart type. CLOSED union; OMITTED = "bar". "bar" is the only value in
     *  v4.1; `line` is an ADDITIVE future value (CHART-LINE), NOT a new node —
     *  so consumers/agents key off `kind`, and a later milestone widens the
     *  union without a wire break. */
    kind?: "bar";
    /** Ordered category→value data. Each point is a SELF-CONTAINED
     *  {label, value} pair (mirroring StatItem) so an agent reads the series
     *  directly with no parallel-array index alignment. Bars render in array
     *  order. */
    points: ChartPoint[];
    /** Optional chart title rendered above the plot. */
    title?: string;
    /** Optional appearance tone from the existing tone axis, mapped to the
     *  theme's --vms-* tone tokens. NO raw color/CSS crosses the wire (D3). */
    tone?: "danger" | "warning" | "success" | "info";
  }

--- .NET (viewmodel-shell-dotnet/ViewModels.cs) ---

  public record ChartPoint(string Label, double Value);

  public record ChartNode(
      IReadOnlyList<ChartPoint> Points,
      [property: JsonIgnore(Condition = JsonIgnoreCondition.WhenWritingNull)] string? Kind = null,
      [property: JsonIgnore(Condition = JsonIgnoreCondition.WhenWritingNull)] string? Title = null,
      [property: JsonIgnore(Condition = JsonIgnoreCondition.WhenWritingNull)] string? Tone = null
  ) : ViewNode;

  // discriminator line in the [JsonPolymorphic] block:
  [JsonDerivedType(typeof(ChartNode), "chart")]

--- Design justifications (the two decisions the operator is signing off) ---

1. POINTS SHAPE — `points: {label,value}[]` (a ChartPoint record), NOT parallel
   `categories:string[]` + `values:number[]`. Chosen because:
   • Agent-legibility (D1): each datum is a self-contained pair; the agent never
     has to zip two arrays or worry about a length/index mismatch. Strictly MORE
     legible than parallel arrays. Mirrors the existing StatItem/StatBarNode
     precedent (a list of paired records), so it is idiomatic to this codebase.
   • Parity byte-identity: string labels round-trip byte-clean across TS/.NET.
     The ONLY numeric-drift risk (TS `number` ↔ .NET `double`) is IDENTICAL
     whether the number sits in `points[].value` or a flat `values[]` — the
     shape does not add risk. It is neutralized by using WHOLE-NUMBER values in
     the parity fixture (12-02): System.Text.Json emits a whole double as `12`
     and JSON.stringify emits `12` — byte-identical. `.NET double` is the correct
     type to mirror TS `number` (ProgressNode uses `int` only because it is a
     0–100 integer; chart values are real magnitudes, so `double`).

2. `kind` DISCRIMINATOR — an OPTIONAL `kind?: "bar"` field (omitted = "bar")
   rather than baking "bar" into the node type. This makes `line` an ADDITIVE
   future union value (CHART-LINE) instead of a new `LineChartNode`, keeping the
   node count flat and the extension purely additive. Only "bar" ships now, so
   the type admits only what is implemented; the renderer treats absent as "bar".

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| server→client wire | The `ChartNode`'s `points` (labels + numeric values), `title`, `tone`, `kind` cross the wire. The browser renders labels/values into a Chart.js bar config. `tone` is mapped through a CLOSED local map to a `--vms-*` token — it is NOT interpolated into a class name or raw CSS, so there is no style/class-injection surface. |
| adapter→chart.js | `chart.js` is dynamically imported inside the adapter; only server-authored view data reaches it (the server already fully controls the tree). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-01 | Tampering | browser.ts chart() rendering point labels/values | accept | Labels are set as Chart.js `data.labels` (canvas-drawn text, not innerHTML), values as numbers — no HTML/CSS injection surface. `tone` is a closed union mapped to a fixed token set; an off-union tone falls to the neutral default, never emits arbitrary CSS. A hostile server already controls the whole view tree; ChartNode adds no new injection capability. |
| T-12-02 | Denial of Service | browser.ts chartInstances lifecycle | mitigate | Instances are keyed and mark-swept every render — a chart dropped from the new tree is `destroy()`'d, so a long session cannot accumulate orphaned Chart instances/canvases. The canvas is reused (not re-created) when the node persists, bounding allocation. Verified by the removal-`destroy()` + redraw-`update()` tests (Task 3). |
| T-12-03 | Information Disclosure | missing optional chart.js dependency | mitigate | A ChartNode with chart.js absent FAILS LOUD (throws with an install hint) rather than silently no-op — the capability-seam rule — so a missing-viz bug is debuggable, not a swallowed blank. |
| T-12-SC | Tampering | npm install of chart.js | mitigate | `chart.js` is a well-known, actively-maintained package (see RESEARCH/memo: chosen over frappe-charts/ApexCharts on maintenance health). It is a v4 pinned optional peer + devDependency. NO other new packages. Verify the installed package is the official `chart.js` on npmjs.com/package/chart.js before relying on it. |
