<objective>
Build the second of three purpose-built human-verification demo apps: **poll + user-action
coexistence**, proving NBA-05 (folding `pollInterval` into the non-blocking lane fixed the
single-mutex poll/user contention). A view with an always-on poll (an artificially slowed
server round trip so overlap with a user click is essentially guaranteed) ticking a counter, plus a
plain blocking "Click me" button incrementing a second counter. The operator clicks during a poll
round trip and confirms the click lands immediately and the poll keeps ticking afterward — the
"today-vs-fixed contrast" the design doc calls for (narrated in 16-04's script, since the actual
pre-Phase-14 buggy mutex no longer exists in this codebase to literally toggle between).

Purpose: give the operator (and 16-04's combined script) a deterministic, always-reproducible
surface for NBA-05 — the poll's own artificial delay (~1.8s) comfortably exceeds its own interval
(~1.2s), so a poll round trip is essentially always in flight, making "click during a poll" trivial
to trigger without precise timing.

Output: a runnable single-process Bun full-stack app at `demo/NonBlockingPoll-bun/` (port 3009),
following the same `Tasks-fullstack-bun` pattern as 16-01. No automated test file — human-verified
in 16-04 per NBA-08's own verification method.
</objective>

From `viewmodel-shell/src/index.ts`:
```typescript
export interface ShellOptions {
  endpoint: string;
  actionEndpoint: string;
  adapter: Adapter;
  /** When set, the shell dispatches a "poll" action at this interval (ms) after every
   *  load/dispatch. NBA-05: poll ALWAYS rides the non-blocking lane — a blocking user
   *  action fired while a poll round trip is in flight is honored, not dropped. */
  pollInterval?: number;
  onLoading?(loading: boolean): void;
  onError?(err: Error): void;
}
export interface StatBarNode { type: "stat-bar"; stats: Array<{ label: string; value: string | number }>; }
export interface ButtonNode { type: "button"; label: string; action: ActionEvent; emphasis?: "primary" | "secondary"; }
```

The server side of the poll contract needs NO special handling beyond a normal `case "poll":` action
branch — the client auto-dispatches `{name:"poll"}` silently on the configured `pollInterval` cadence;
the server just returns a fresh `{vm, state}` like any other action. (`nextPollIn` on the response is
OPTIONAL — omitting it lets the client's own `ShellOptions.pollInterval` keep driving the cadence,
which is what this demo relies on; no need to set it explicitly.)

From `viewmodel-shell/src/server.ts`: same `createAction`/`ActionPayload`/`UnknownActionError`/
`createAgentSkillHandler` surface as 16-01's `<interfaces>` block — reuse identically.
