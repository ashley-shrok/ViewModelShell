<objective>
Build the third of three purpose-built human-verification demo apps: **out-of-order / staleness
discard**, proving NBA-03 (a client-side sequence counter discards a stale/out-of-order response
rather than letting it clobber a newer render). Two buttons: one fires a SLOW `blocking:false`
background update (~3s artificial delay); the other fires a FAST default-blocking user update
(~150ms). Triggering slow-then-immediately-fast and observing the final value stays "fast" — even
after waiting long enough for the slow response to actually arrive — is the deterministic,
reproducible proof this scenario needs. (Coalescing already guarantees at most one non-blocking
round trip is ever in flight per lane, so the genuinely observable out-of-order race in this
codebase is cross-lane: a background/non-blocking response racing against a blocking one — see the
design doc's Epoch section on the blocking-lane-always-applies rule.)

Purpose: give the operator (and 16-04's combined script) a fully deterministic staleness-discard
demo with no timing precision required from the human — the 3-second gap between triggers is wide
enough that "click slow, then click fast within a couple of seconds" always reproduces the race.

Output: a runnable single-process Bun full-stack app at `demo/NonBlockingStaleness-bun/` (port 3010),
following the same `Tasks-fullstack-bun` pattern as 16-01/16-02. No automated test file — human-
verified in 16-04 per NBA-08's own verification method.
</objective>

<!-- The exact client-side epoch mechanics this demo's TWO buttons are designed to exercise
     (already shipped in Phases 14-15; read for context, do not modify). -->

From `.planning/design/non-blocking-actions.md`'s "Epoch — client-side, off the wire" section
(paraphrased, the load-bearing rule this demo's shape depends on): a monotonic client-side sequence
counter is stamped on every dispatch. **A blocking response always applies unconditionally the
moment it arrives** (it can never be superseded — at most one blocking dispatch is ever in flight).
**A non-blocking (background) response is discarded if a strictly newer response has already
applied** (`seq >= appliedSeq` gate). So: fire the SLOW non-blocking trigger first (seq 1, in
flight ~3s) — while it is still in flight, fire the FAST blocking trigger (seq 2, resolves in
~150ms and applies immediately, advancing `appliedSeq` to 2). When the slow response (seq 1) finally
arrives, `1 < appliedSeq(2)` so it is discarded — the displayed value never reverts.

From `viewmodel-shell/src/index.ts`: `ButtonNode { type: "button"; label: string; action: ActionEvent }`
where `ActionEvent.blocking?: boolean` (omitted = true). From `viewmodel-shell/src/server.ts`: the
same `createAction`/`ActionPayload`/`UnknownActionError`/`createAgentSkillHandler` surface used in
16-01/16-02 — reuse identically.
</interfaces>
