**Bun full-stack demo app (port 3009) with an auto-dispatched, artificially-slowed poll (~1.8s) racing a 1.2s client cadence against a plain blocking "Click me" button, proving NBA-05 poll/user-action coexistence.**

## Accomplishments
- Scaffolded `demo/NonBlockingPoll-bun/` mirroring the `Tasks-fullstack-bun` / `NonBlockingActionBar-bun` single-process pattern (zero custom CSS, agent-discoverability meta tag, dark-purple theme).
- Wired the client with `pollInterval: 1200` so `shell.load()` kicks off an automatic `{name:"poll"}` dispatch every 1.2s via the non-blocking lane.

- Verified end-to-end: `tsc --noEmit` clean, `vite build` succeeds, and a live `bun run server.ts` smoke test confirmed `GET /api/poll` returns `{vm, state}` with `pollTicks`/`userClicks` and `POST /api/poll/action` correctly increments `userClicks` on `increment-click`.

## Decisions Made
- Poll delay (1800ms) intentionally exceeds the client's `pollInterval` (1200ms), per the plan's explicit design intent, so overlap between a poll round trip and a user click is essentially guaranteed rather than timing-dependent.
- `increment-click`'s action omits the `blocking` field entirely (defaults to blocking) — this is the scenario's whole point: a normal blocking action must still land instantly even while a non-blocking poll is mid-flight.
- No `nextPollIn` set on any response — cadence is fully client-driven via `ShellOptions.pollInterval`, matching the plan's `<interfaces>` guidance and avoiding any redundant/conflicting cadence source.
