<objective>
Build the first of three purpose-built human-verification demo apps for the v4.2 Non-Blocking
Actions milestone: **selection → live server-computed action bar**, the PBMInvoices shape named in
the design doc. A table of invoice rows, each with a selection checkbox; an "Approve Selected" /
"Reject Selected" action bar whose enabled/disabled state is recomputed SERVER-SIDE after every
checkbox toggle via a `blocking:false` round trip — never computed client-side. This demo, together
with 16-02 and 16-03, is what the operator runs and signs off against in 16-04 (NBA-08).

Purpose: give the operator (and the combined script in 16-04) a real, clickable surface proving
NBA-06 (optimistic check + echo-back + coalescing never reverts a rapid toggle) and the server
re-validation / `rejected` envelope path (a click that races a stale action-bar rendering into
invalidity is rejected, not silently applied).

Output: a runnable single-process Bun full-stack app at `demo/NonBlockingActionBar-bun/` (port 3008),
following the `Tasks-fullstack-bun` pattern — one `Bun.serve` serving both the Vite-built client and
the `/api/actionbar` wire. No automated test file is added (per the phase's test_gate: this is a
concurrency/timing feature verified by human review in 16-04, not asserted by vitest); the plan's
own acceptance criteria are static/structural checks (grep, `tsc --noEmit`, a manual curl smoke test).
</objective>

```typescript
export interface ActionEvent {
  name: string;
  files?: Record<string, File>;
  /** Optional; omitted = true (blocking). false = non-blocking lane. Client-side only —
   *  NEVER rides the `_action` POST payload; the server only ever sees `{name}`. */
  blocking?: boolean;
}

export interface CheckboxNode {
  type: "checkbox";
  name: string;
  bind: string;               // e.g. "selectedIds.3"
  label?: string;
  action?: ActionEvent;       // dispatched immediately on change
}

export interface ButtonNode {
  type: "button";
  label: string;
  action: ActionEvent;
  emphasis?: "primary" | "secondary";
  tone?: "danger" | "warning" | "success" | "info";
  disabled?: boolean;         // server-owned; renderer will NOT dispatch on click when true
}

export interface TableColumn {
  key: string;
  label: string;
}

export interface TableRow {
  id?: string;
  cells: Record<string, string>;
  actions?: (ButtonNode | CheckboxNode)[];  // CheckboxNodes render in a LEADING column
  state?: string;              // e.g. "done" — lifecycle, not severity
  tone?: "danger" | "warning" | "success" | "info";
}

export interface TableNode {
  type: "table";
  columns: TableColumn[];
  rows: TableRow[];
}
```

From `viewmodel-shell/src/server.ts` (`@ashley-shrok/viewmodel-shell/server` subpath):
```typescript
export interface ActionPayload<TState> { name: string; state: TState; files?: Record<string, File>; }
export interface ErrorEntry { message: string; path?: string; code?: string; }
export interface ShellRejection { violations: ErrorEntry[]; }
export function shellRejection(violations: ErrorEntry[]): { rejected: ShellRejection };
export function createAction<TState>(
  handler: (payload: ActionPayload<TState>) => Promise<ShellResponseBody<TState>> | ShellResponseBody<TState>,
  options?: { currentBuild?: string },
): (request: Request) => Promise<Response>;
export class UnknownActionError extends Error { constructor(actionName: string); }
export function validateActionNames(vm: ViewNode): void;   // THROWS if two non-form nodes share an action name
export function createAgentSkillHandler(opts?: { appPreamble?: string }): (request: Request) => Response;
```

**Critical constraint (verified in `server.ts`'s `validateActionNames`):** two dispatch-bearing nodes
OUTSIDE a shared `FormNode` that fire the SAME action name throw at response-build time (`createAction`
calls `validateActionNames` on every response). Per-row checkboxes are NOT inside a form here, so
**each row's checkbox action name must embed the row id** — e.g. `recompute-3`, `recompute-4` — never
a single shared name like `recompute`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client `_state`/`_action` POST → `server.ts` action handler | The client resubmits the full state blob (including `selectedIds`) on every dispatch — standard VMS "server never trusts client state for correctness" model. This demo's whole point is that the server RE-DERIVES row eligibility from its own `rows` array (server-owned truth), never from anything the client could have tampered with. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-01 | Tampering | `approve-selected` / `reject-selected` handlers | mitigate | Row `status` (locked/approved/rejected/pending) is read from the SERVER's `state.rows` array, never trusted from a client-submitted field describing row content — a client can only submit which ids are selected (`selectedIds`), not what those rows' statuses are, so a tampered client cannot forge eligibility. |
| T-16-02 | Denial of Service | the `recompute-*` artificial `sleep(750)` delay | accept | Fixed demo-only constant, not attacker-influenceable; not a production code path (this app ships nowhere but a local/tailnet demo run). |
| T-16-03 | Information Disclosure | the whole app (no auth) | accept | Local demo app with no real data, matching every other `-bun` demo's no-auth posture (see `demo/Tasks-fullstack-bun`); served only on the tailnet per the phase's standing convention, never publicly. |
| T-16-SC | Tampering | npm/bun package installs | accept | Zero NEW packages: `package.json` mirrors `Tasks-fullstack-bun`'s already-vetted dependency set exactly (`link:`-protocol local `@ashley-shrok/viewmodel-shell` + `@types/bun` + `typescript` + `vite`, all already present elsewhere in this repo's lockfiles) — no external package legitimacy gate applies. |
</threat_model>
