<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| backend → wire JSON | The server serializes `PageNode`/`SectionNode` (incl. the new `layout` field) into the JSON wire response consumed by the browser adapter |
| wire JSON → renderer | The browser adapter deserializes the wire and, in Wave 2, maps `layout` into a CSS class name |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering / Injection | `layout` value flowing toward a CSS class name | mitigate (by-design, closed union) | The TS type is a CLOSED union (`"stack" \| "split" \| "cards"`), not an open `string`. The Wave-2 renderer (Plan 02) emits a class ONLY via an explicit `n.layout === "split" || n.layout === "cards"` equality guard — an arbitrary attacker-supplied string can never be reflected into `className`. This plan only adds the field; the guard is the actual mitigation, asserted in Plan 02. No `high`/`medium` residual. |
| T-04-02 | Information Disclosure | new optional `string?` on already-deserialized records | accept | `Layout` is an optional field on `PageNode`/`SectionNode` records that are ALREADY deserialized by the existing registered JSON pipeline (lines 81-97). It introduces NO new deserialization-of-untrusted-input surface — same parser, same already-trusted record graph. Field is non-PII layout intent. No residual risk. |

**Severity conclusion:** No `high` or `medium` threats. The only security-relevant
consideration — reflecting an attacker-controlled string into a CSS class — is
*mitigated by design* via the closed-union type plus the closed-equality renderer
guard (Plan 02). This plan adds zero new parsing/network/file/exec surface.
</threat_model>
