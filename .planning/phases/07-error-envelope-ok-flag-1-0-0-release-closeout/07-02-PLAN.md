<objective>
Rewire the shell's response-handling paths so the framework-set `ok` flag drives a single, uniform branch: parse the body unconditionally, check `body.ok`, and either route through `processResponse` (success) or construct a `VmsActionError` and hand it to the existing `onError` callback (failure). Add the new exported `VmsActionError` class — sole new public API surface on the client side. Extend the `ShellResponse` interface with an optional `ok?: boolean` so existing `shell.push()` consumers don't break.

Purpose: this is the consumer-facing half of the envelope contract. Without this plan, the new envelope responses from Plan 01 land in the client as opaque 4xx/5xx errors that lose the structured payload (today's `if (!res.ok) throw new Error("Action '...' failed: 4xx")` paths at lines 450 and 523 swallow the body). Per D-13/D-14, apps that wired `onError` for fetch failures keep working unchanged; apps that want the structured payload do `if (err instanceof VmsActionError) { ... err.errors ... }`.

Output: a single point of truth for "the framework signaled failure" on the client side, accessible from every response path (GET load, POST dispatch, external push), with no new callbacks (only `VmsActionError` riding the existing `onError` surface).
</objective>
