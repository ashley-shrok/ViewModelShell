<objective>
Add framework-side enforcement of the action-name uniqueness rule from CONTEXT.md: "one action name = one operation." When a server builds a ViewNode tree, the framework walks it and errors if two dispatch-bearing nodes declare the same action name but represent semantically distinct operations.

Purpose: Make the rule structural — a tree-build-time check, not a runtime guess. Catches the "I accidentally reused 'save' for two different operations" bug class before it reaches the wire.
Output: Validator functions in both backends (TS + .NET), invoked automatically before the server emits the response. Distinct error message names the violating action name and the two nodes' rough locations.
</objective>

  <files>viewmodel-shell/src/server.ts</files>

  <action>
Add a `validateActionNames(vm: ViewNode): void` function to `viewmodel-shell/src/server.ts`, placed right after the `ViewNode` re-export (after the `export * from "./index.js"` line). The function:

1. Walks the entire ViewNode tree recursively.
2. Collects every dispatch-bearing action: from `ButtonNode.action`, `FormNode.submitAction`, `FormNode.buttons[].action`, `FieldNode.action` (optional), `CheckboxNode.action` (optional), `TabsNode.tabs[].action`, `TableNode.sortActions` values, `TableNode.pagination.prevAction`/`nextAction`, `TableRow.actions[].action`, `ModalNode.dismissAction`.
3. Groups by action name. For each group with 2+ entries, the rule is: **same name is allowed only if the two nodes share the same enclosing form context** (i.e. two Save buttons on the same form, the canonical valid duplicate). Otherwise: throw.

**Heuristic (the only one that closes the most important error case):** Walk the tree tracking an enclosing-form stack. Two nodes are considered "same operation" iff **both have a non-null enclosing FormNode AND share the same FormNode reference**. Outside a form, ANY duplicate action name is a violation — there's no notion of "same operation" between two top-level buttons, because the most common bug is per-row buttons that forgot to include the row ID in the name (`delete-row` repeated on every row, instead of `delete-row-42` / `delete-row-43`). Catching that bug is exactly why this check exists; the heuristic must not have a loophole that swallows it.

Concretely:
- Two ButtonNodes inside the same FormNode firing `save-ticket-42`: PASS (canonical valid duplicate — top/bottom save).
- Two ButtonNodes in different FormNodes firing `submit`: FAIL.
- Two ButtonNodes at the page level (no enclosing form) firing `delete`: FAIL (this is the common bug — apps must rename to `delete-row-42` / `delete-row-43`).
- A ButtonNode in a form and another at page level firing `save`: FAIL.

Error shape: `throw new Error(\`Duplicate action name '${name}' dispatched from semantically distinct nodes. Each action name must name exactly one operation. Either rename one of the nodes (e.g. '${name}-X' / '${name}-Y') or move them into the same surrounding form if they are intended to fire the same operation.\`);`

The function is exported. Invocation: at the end of `createAction`'s handler wrapper, after the user's handler returns, if `result.vm` is non-null, call `validateActionNames(result.vm)` BEFORE serializing. On throw, return a 500 with the error message (similar to how the existing parse-error path returns 400 — see lines 138-143).

Add this clause to `createAction`:
```typescript
if (result.vm) {
  try {
    validateActionNames(result.vm);
  } catch (err) {
    return new Response(JSON.stringify({ error: (err as Error).message }), {
      status: 500,
      headers: { "Content-Type": "application/json" },
    });
  }
}
```

Also call `validateActionNames` from any helper an app uses to build the initial GET response — but the TS server subpath doesn't ship one of those (apps build their own GET handlers). Document this in the JSDoc: "Call `validateActionNames(vm)` yourself in your GET handler before returning if you want the same protection at initial-load time."

Create a new vitest file `viewmodel-shell/src/tree-walker.test.ts`. It is a pure JS test (no jsdom needed) — it imports `validateActionNames` from `./server.js` and constructs ViewNode trees as plain objects.

Test cases:

1. **`passes: unique action names across distinct buttons`** — Build a `PageNode` with two `ButtonNode`s having different action names. Call `validateActionNames(page)`. Expect no throw.

2. **`passes: same action name, same form (canonical valid duplicate)`** — Build a `PageNode` with one `FormNode` containing two `ButtonNode`s (one labeled "Save top", one labeled "Save bottom"), both with `action: { name: "save-ticket-42" }`. Call `validateActionNames`. Expect no throw.

3. **`throws: same action name across different forms`** — Build a `PageNode` with two `FormNode`s, each containing a `ButtonNode` with `action: { name: "submit" }`. Call `validateActionNames`. Expect throw with message containing `Duplicate action name 'submit'`.

4. **`throws: same action name between a top-level button and a form-internal button`** — `PageNode` with one `ButtonNode` (action: "delete") at the page level, and one `FormNode` containing a `ButtonNode` with action: "delete". Expect throw.

5. **`passes: per-tab unique action names`** — Build a `TabsNode` with three tabs, action names `select-tab-a`, `select-tab-b`, `select-tab-c`. Bind paths trivially set. No throw.

6. **`throws: TabsNode with duplicated tab action names`** — Two tabs declaring the same action name in their `tabs[].action.name`. Throw.

7. **`passes: per-row actions on a TableNode use unique action names`** — Build a TableNode with two rows, each row's `actions: [{ type: "button", action: { name: "delete-row-1" } }]` and `delete-row-2`. No throw.

8. **`throws: per-row actions duplicate`** — Two rows both declare `delete-row` (no per-row identity in the name). Throw.

9. **`passes: TableNode sortActions, prevAction/nextAction, filterAction(implicit) all uniquely named`** — full TableNode with sortActions per column (`sort-by-title`, `sort-by-date`), pagination prev/next (`page-prev`, `page-next`). No throw.

10. **`throws: two top-level buttons (no enclosing form) share an action name`** — PageNode with two ButtonNodes (no FormNode wrapping either), both with `action: { name: "delete" }`. Throw. This is the common per-row-buttons-missing-row-id bug pattern; the strict outside-form heuristic must catch it.

Each test uses `expect(() => validateActionNames(tree)).toThrow(/Duplicate action name/)` or `.not.toThrow()`.

The test file should have a `describe("validateActionNames", () => { ... })` block grouping these.
