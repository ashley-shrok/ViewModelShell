**Demo overview:** Tasks renders a list of TODO items with an add-task form, per-row toggle checkboxes, per-row delete buttons, and a filter tab strip (all / active / completed).

**Tasks-bun (`demo/Tasks-bun/server.ts`):**

1. Update `TasksState` to add a `draftTitle: string` slot (default `""`) — this is where the add-task input's typed value lives now. (If the existing state has a `newTaskTitle` or similar, reuse it; otherwise add `draftTitle`.)

2. Update `buildVm`:
   - The add-task FormNode's FieldNode for the title gets `bind: "draftTitle"`. The FieldNode no longer carries a `value` prop.
   - The submit ButtonNode keeps its `action: { name: "add" }`. Since there's only one add operation, the action name doesn't need to encode anything.
   - Filter tabs: each tab gets its own action name — `select-tab-all`, `select-tab-active`, `select-tab-completed`. The TabsNode adds `bind: "filter"` so the renderer writes the clicked tab value to state.filter automatically; the action also fires (server may need to do anything else after the filter switch).
   - Per-row checkbox: bound to `items.${i}.completed` where `i` is the row's array index, action name `toggle-row-${item.id}`.
   - Per-row delete button: `action: { name: \`delete-row-${item.id}\` }`.

3. Update `actionHandler`:
   - The handler now switches on `payload.name` only — no `payload.context`. Remove the `ctx`, `str`, `bool` helpers.
   - `"add"`: reads `state.draftTitle`, creates a new TaskRecord, appends to `state.items`, resets `state.draftTitle = ""`. (Resetting the draft is part of the post-submit UX.)
   - `"toggle-row-XXX"`: the action name encodes the row ID — pattern-match with `if (payload.name.startsWith("toggle-row-")) { const id = payload.name.slice("toggle-row-".length); ... }`. Then read the current completed state from `state.items` (find by id), and either flip it (toggle semantics) OR if the renderer already wrote the new value to state via the bind path on the checkbox, just keep the new value as-is. The bind+action pair means the new value is already in state when the action fires; the server's job is just to persist it. So: no flip needed — just re-render the same state.
   - `"delete-row-XXX"`: same pattern; extract id, filter from state.items.
   - `"select-tab-XXX"`: extract the tab value from the action name suffix. State.filter is already written by the renderer (because of the bind). Server just acknowledges with a re-render.
   - Default: throw (and let framework error envelope handle it — Phase 7).

4. Update existing unit tests (in `Tasks-bun/server.test.ts` if any exists) similarly.

**Showcase** is the all-presets, all-themes gallery demo. It is fundamentally different from the other demos: it has **no backend** and does NOT use `ViewModelShell`. It calls `adapter.render(buildVm(), handle)` directly against a local `BrowserAdapter`, and its action handler reads `action.context` extensively. This migration has structural impact, not just type cosmetics.

Concrete migration steps for Showcase:

1. **Local state object.** Showcase already maintains a top-level `state` object (or equivalent — read the existing main.ts to see the exact name). Audit every place its action handler currently does `ctx.X` reads; for each, the value must live in `state` at a known path so the renderer writes it there via the bind seam.

2. **stateAccess seam without a shell.** Plan 03 Task 2 made `Adapter.render` take a required third `stateAccess` parameter. Showcase calls `adapter.render(vm, handle)` directly — that signature now needs three args. Construct a minimal stateAccess inline using the same `readPath`/`writePath` helpers Plan 03 Task 1 added to `viewmodel-shell/src/index.ts` (export them from `viewmodel-shell` if not already; the Plan 03 SUMMARY should note whether they're exported. If not, copy them inline in Showcase's main.ts — they're 20 lines). The closure is:
   ```ts
   const stateAccess = {
     read: (path: string) => readPath(state, path),
     write: (path: string, value: unknown) => { state = writePath(state, path, value); rerender(); },
   };
   adapter.render(buildVm(state), handle, stateAccess);
   ```
   The `rerender()` is whatever Showcase already uses to re-call `render` after action handlers mutate state. Adding it to `write` means every keystroke triggers a re-render — that matches Showcase's existing manual rerender pattern; if performance matters, debounce later.

3. **Action handler — remove every `ctx.X` read.** Current Showcase handler reads from `action.context`:
   - `state.selectedItemId = String(ctx.id)` → instead, the action name encodes the id. Rewrite as: `if (action.name.startsWith("list-detail:select-")) { state.selectedItemId = action.name.slice("list-detail:select-".length); }`.
   - `state.selectedTab = String(ctx.value)` → tab action names per-tab: `select-tab-<value>`. The TabsNode bind also writes state.selectedTab automatically; the action handler may not even need to set it.
   - `state.sortColumn = String(ctx.column); state.sortDirection = ...` → with `sortBind: "sortIntent"`, the renderer writes `{column, direction}` to `state.sortIntent` before dispatching. The handler reads `state.sortIntent`.
   - `state.filters = { ...state.filters, ...(ctx.filters as Record<string,string>) }` → with `filterBinds` per column, the renderer writes per-column values to `state.filters.<col-key>`. Handler does nothing — the writes already happened.

4. **Per-row buttons:** Showcase's list/detail archetype has per-row "Select" buttons currently using `context: { id: r.id }`. Rewrite each row's button action name to `list-detail:select-${r.id}`. Apply uniqueness rule per Plan 02 (Showcase's tree-builds happen client-side, so `validateActionNames` from Plan 02 doesn't run unless Showcase explicitly calls it; recommended: call `validateActionNames(buildVm(state))` once per render to catch development-time bugs).

5. **TabsNode reshape:** the Showcase archetype-switcher uses TabsNode; under the new shape (Plan 01) each tab carries its own action. Update the tab construction: `tabs: [{value: "dashboard", label: "Dashboard", action: { name: "show:dashboard" }}, ...]`.

6. **TableNode in the list/detail archetype:** if it currently uses sortAction/filterAction/selection — apply the same patterns as the workflow demos (Tasks 1-3). Selection (if any) becomes per-row CheckboxNodes bound to state slots.

7. **Visual regression:** the point of Showcase is visual demonstration. After all changes, run `cd demo/Showcase/frontend && npm run dev`, open the page in a browser, click through every archetype, confirm visuals are identical to pre-migration. The wire change doesn't affect rendering, only data flow — anything visual that breaks is a migration bug.
