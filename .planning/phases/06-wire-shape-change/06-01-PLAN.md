<objective>
Update the ViewNode type system (TS + .NET) and the action-payload parsing layer (TS + .NET) to support the new wire contract: every input node declares a `bind` path into state, and the dispatched action carries no `context` payload at all. This plan is purely types + parser ā€” the renderer (Plan 03) and tree-build uniqueness check (Plan 02) consume the new types but live in their own plans.

Purpose: Land the wire-format shape changes that everything else in Phase 6 depends on. Both backends must agree byte-for-byte on the new shape from day one.
Output: A TS `ViewNode` union and a .NET record set that both declare `bind` on every input-bearing node, no `context` field anywhere on the wire, and a TableNode without `selection` / per-cell `Action` plumbing for selection. ActionPayload reads `{name, state}` only.
</objective>

Edit `viewmodel-shell/src/index.ts`:

1. **`ActionEvent` (lines 3-7):** Remove the `context?: Record<string, unknown>` field. The new shape is just `{ name: string; files?: Record<string, File> }`. Note: `files` stays ā€” it's the multipart side channel for file uploads.

2. **`FieldNode` (lines 148-169):** Add `bind: string;` as a required field. This is the path into state where the field's value is read/written (e.g. `"fields.title"`, `"filter"`, `"rows.42.completed"`). Remove the `value?: string` field ā€” the value now comes from state at the bind path, not from a separate prop. The existing `action?: ActionEvent` field stays (auto-dispatch on Enter for text-like inputs is preserved per CONTEXT.md decision).

3. **`CheckboxNode` (lines 171-178):** Add `bind: string;` as a required field. Remove `checked: boolean` (read from state at bind path instead). The existing `action?: ActionEvent` field stays (auto-dispatch on change is preserved).

4. **`TabsNode` (lines 226-232):** Final shape ā€” add `bind: string;` as a required field, **remove the top-level `action: ActionEvent` field**, and reshape `tabs` so each tab carries its own action: `tabs: Array<{ value: string; label: string; action: ActionEvent }>`. The `selected: string` field stays (it's a copy for the renderer to highlight the current tab without walking state; server fills it from the bound state value). Each tab's action name must be unique (e.g. `select-tab-pending`, `select-tab-active`, `select-tab-done`); Plan 02's `validateActionNames` enforces uniqueness.

5. **`TableColumn` (lines 252-262):** Keep this record unchanged. Sort/filter live in app state and are bound separately.

6. **`TableRow` (lines 264-269):** Replace `action?: ActionEvent` with `actions?: ButtonNode[]`. A row's per-row action(s) are now full buttons with unique action names. This makes the wire honest (the row carries the same button shape any other place does) and supports multiple per-row actions naturally.

7. **`TableSelection` (lines 271-286):** Delete this interface entirely.

8. **`TablePagination` (lines 288-299):** Remove the `action: ActionEvent` field. Pagination state (current page) lives in state and is bound via TableNode's `paginationBind` (see #9). Page-change buttons are rendered by the renderer with unique action names the server includes on the pagination record ā€” OR a simpler shape: keep `page`, `pageSize`, `totalRows` and add `prevAction?: ActionEvent`, `nextAction?: ActionEvent` (each with its own unique action name). Apps name them like `page-prev`, `page-next`.

9. **`TableNode` (lines 301-323):** Remove `sortColumn?`, `sortDirection?`, `sortAction?`, `selection?`. **Keep `filterAction?: ActionEvent`** ā€” one filter-dispatch action per table is the simplest shape and matches the renderer (Plan 03 fires this when the user submits the filter form, with per-column filter VALUES already in state at the bound paths). Add `sortBind?: string` (the renderer writes the current sort intent ā€” `{column, direction}` ā€” to this path), `filterBinds?: Record<string, string>` (per-column filter input bind paths), `paginationBind?: string` (the renderer writes the target page number to this path before dispatch). Sort header clicks dispatch a unique action name per column; add `sortActions?: Record<string, ActionEvent>` keyed by column key so the renderer knows which action to fire per column.

10. **`FormNode` (lines 126-146):** Remove `submitAction?.context` usage assumption ā€” the field is still `submitAction?: ActionEvent` but ActionEvent no longer has context. Same for `buttons?: ButtonNode[]` ā€” each button has an action with no context. No structural change to FormNode beyond what ActionEvent's shape change implies.

11. **Hidden `inputType: "hidden"` on FieldNode (line 154):** Hidden fields still exist but their value comes from state at the bind path, not from a `value` prop. No special case needed.

12. **Update `ShellResponse` (lines 378-397):** No change needed ā€” this is the server-to-client envelope; this plan only touches the client-to-server side.

Write the JSDoc above each new `bind` field to be a one-liner stating: "Path into state where this input reads its current value and writes user changes (e.g. `fields.title`)."

When you finish, the file must `tsc --noEmit` cleanly. Some downstream consumers in `browser.ts` will type-error until Plan 03 lands ā€” that's expected. Add `// @ts-expect-error WIRE-06 (Plan 03) will update browser.ts` to the imports in browser.ts only if needed to keep the broader build green for testing this plan in isolation.
