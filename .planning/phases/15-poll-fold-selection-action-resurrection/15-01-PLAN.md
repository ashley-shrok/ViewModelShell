<objective>
Close the two remaining gaps in the Phase-14 non-blocking dispatch core so the ROADMAP's Phase 15
Success Criteria #1 and #2 are actually true, not just theoretically implied.

**NBA-05 (poll-fold) — verify-then-document, no code-semantics change.** Phase 14 already made
`schedulePoll()`'s `dispatch({name:"poll"}, true)` call route through the non-blocking lane (because
`nonBlocking = silent || action.blocking === false` is `true` whenever `silent` is `true`) — so a
poll round trip already coexists with a blocking user action instead of contending for the same
mutex, and this exact coexistence is already exercised (via manual `dispatch({name:"poll"}, true)`
calls, not the real timer) by the Phase-14 gap-closure tests CR-01/CR-02 in
`nonblocking-dispatch.test.ts`. **The remaining delta is: (a) tests that drive this through the REAL
`ShellOptions.pollInterval` → `setTimeout` → automatic dispatch path** (not manual `dispatch(poll,
true)` calls) so the ROADMAP Success Criterion is proven end-to-end, not just at the unit-internals
level, and **(b) explicit code-comment documentation** on `schedulePoll`/`ShellOptions.pollInterval`
stating that poll intentionally rides the non-blocking lane, citing NBA-05 and the design doc — no
change to `schedulePoll`'s actual logic is needed or made.

**NBA-06 (selection.action resurrection) — a REAL, minimal client-side fix is required.** Tracing the
exact interleaving of a rapid DOUBLE-toggle on the SAME checkbox through the current Phase-14 code
reveals a genuine unclosed gap: toggle A fires (seq N, blocking:false, in flight); toggle B (the
user's very next click, un-toggling A) coalesces into `pendingNonBlockingRefire` because
`nonBlockingInFlight` is already true. When A's response arrives, it is NOT stale by the existing
`seq >= appliedSeq` epoch rule (it's the only response outstanding) — so it applies via
`processResponse`, which does a WHOLE-STATE REPLACE (`this.currentState = body.state`). Because A's
response necessarily echoes back the state as of A's OWN send time (before B's local optimistic
write happened), applying it overwrites B's not-yet-sent local write with A's older value — the
"box checked, then unchecked" ends up rendering CHECKED, and — because the coalesced refire (B)
captures `this.currentState` fresh AT ITS OWN FIRE TIME, which is now A's stale echo — B's own
request is fired with the WRONG (A's) value too, permanently losing the user's second toggle. This
is exactly the 0.15.0 `selection.action` failure mode NBA-06 exists to close. The fix: extend the
non-blocking apply gate so a response is discarded not only when a NEWER response has already
applied (existing `seq >= appliedSeq`), but ALSO whenever a coalesced re-fire is already queued
(`pendingNonBlockingRefire !== null`) at the moment this response is ready to apply — because a
strictly newer round trip is guaranteed to fire immediately after and supersede it. This requires NO
wire change (pure client-side, reuses the existing `pendingNonBlockingRefire` field from Phase 14)
and does not touch the blocking lane (CR-02's "always applies unconditionally" rule is untouched).

Purpose: make `pollInterval` and per-checkbox/table-selection `blocking:false` refreshes both
actually safe to ship, closing the exact two gaps the ROADMAP's Phase 15 Success Criteria describe.
Output: one corrected client-side dispatch-loop rule (NBA-06), doc clarifications (NBA-05), and
three new/extended test files proving both end-to-end.
</objective>

```typescript
private schedulePoll(nextPollIn?: number): void {
  const delay = nextPollIn ?? this.options.pollInterval;
  if (delay == null) return;
  if (this.pollTimer) clearTimeout(this.pollTimer);
  this.pollTimer = setTimeout(() => {
    this.pollTimer = null;
    this.dispatch({ name: "poll" }, true);
  }, delay);
}
```
