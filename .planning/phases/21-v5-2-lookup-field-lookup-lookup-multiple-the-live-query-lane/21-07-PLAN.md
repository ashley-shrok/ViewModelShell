<objective>
Give the `@experimental` TUI a legible degradation for both lookup inputTypes, and prove the lookup wire serializes byte-identically across all three backends via the FeatureProbe.

Purpose: the cross-backend rule requires every target to at least degrade sensibly, and the parity suite is what actually enforces no-drift between the TS and .NET type sources. The probe is also the phase's cheapest insurance against the highest-cost miss: a uniquely-named `searchAction` in the probe tree is what makes a validator that failed to descend fail the BUILD rather than 500 on a real controller later.

Output: a `lookup`/`lookup-multiple` branch in `tui.tsx`; the lookup view-shape added to `buildVm` in all three FeatureProbe backends; an appended `$comment` clause; parity green.

🚨 **CORRECTING THE CONTEXT AND THE ROADMAP.** Both say "new FeatureProbe fixtures per inputType" / "a new FeatureProbe fixture per inputType". **The shipped v5.1 pattern is NEITHER.** There is **no new fixture file** and **no `backends.json` change** — `dotnet-probe`/`bun-probe`/`node-probe` already register `"fixtures": ["feature-probe", "feature-probe-envelope"]`. The pattern is: extend `buildVm` in the three probe backends, let the EXISTING GET steps capture and byte-diff it, and append a clause to the `$comment`. A new POST step is added ONLY when a feature changes a RESPONSE, not a view-shape — a lookup is a view-shape. **Per 21-PATTERNS.md §7 this is the single lowest-risk task in the phase; do not turn it into the highest by inventing a fixture.**
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| .NET wire ↔ TS wire | The parity harness is the only automated gate proving the two backends emit identical bytes |
| view tree → both action-name walkers | The probe's uniquely-named searchAction is the only automated proof that BOTH walkers descend |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-21-24 | Repudiation | cross-backend wire drift (null/false-vs-absent, `type` naming, array-vs-object) | mitigate | The probe covers the full omitted-vs-present matrix and the existing GET steps byte-diff .NET/bun/node; a drifted ignore condition or a mis-named `type` fails the run |
| T-21-25 | Tampering | a validator that silently fails to descend into `searchAction` | mitigate | `lookup-search-probe` is a unique action name inside the probe tree — a walker that skips it produces a differing tree/validation outcome and fails the build (the v5.1 `nav-crumb-probe` precedent) |
| T-21-26 | Denial of Service | the TUI hitting UnsupportedView on a lookup | mitigate | An explicit branch degrades to a bound text input; the header contract documents it; the TUI is `@experimental` and honest degradation is the stated philosophy |

No third-party package installs in this plan → no legitimacy gate needed.
</threat_model>
