<objective>
Declare the lookup wire surface in the TypeScript backend (the source of truth the .NET twin mirrors and the renderer consumes), and teach the TS tree-validator that `searchAction` is a dispatch-bearing field whose name must be unique.

Purpose: nothing else in this phase can be built against a shape that doesn't exist. And the direction invariant (D1) is a property of the TYPE and its TSDoc before it is a property of any renderer — if the type doesn't say "the label is server→client only, never resolved from `candidates`", the next implementer resolves it from `candidates` and ships the Ant Design failure silently.

<decision_records>
Three of design §8's OPEN points are settled HERE because they are wire-surface questions. Record each in the SUMMARY with its reasoning.

**OPEN-1 — `textArrangement` in v1 or fast-follow? ⇒ DECIDED: FAST-FOLLOW. Do NOT add the field in this phase.**
Reasoning (adopting the design's recorded lean): the enum is purely additive (`"text"` default / `"text-id"` / `"id-text"`), so shipping it later costs nothing and breaks nothing. v1's surface is already large — two inputTypes, five new fields, a new dispatch cadence, and the chips a11y contract. Keeping the v1 surface honest means shipping only what the headline proofs need, and no success criterion depends on rendering the id alongside the label. **Add a TSDoc note on the node naming `textArrangement` as a designed, deferred fast-follow** so the next implementer finds the decision instead of re-deriving it. `TextSeparate` stays permanently out of scope (21-CONTEXT).

**OPEN-3 — is server-side "is this id one of mine?" enough to distinguish picked-from-invented? ⇒ VERIFY, then decide. Recorded lean: server-decidable is enough; NO `__isNew__` wire marker.**
Reasoning to verify against the code, not assume (D3's whole point is explicitness, so this must be checked rather than defaulted): the D1 direction invariant means the client only ever sends `bind` — an id, or an array of ids. The server, which produced every candidate it ever offered, can test membership against its own data. react-select needs `__isNew__` because it is client-only and has no authority to ask. **The verification step is concrete: confirm that with `allowCustom:true` there is NO wire path by which the client communicates provenance, and that none is needed — i.e. that a bare `bind` value plus the server's own id space is sufficient to classify.** If the check shows the server genuinely cannot classify (e.g. an id space where an invented value could collide with a real id), STOP and surface it — that would mean D3 needs a wire marker after all, which is a design change, not an implementation choice. Record the outcome either way. Note the explicitness D3 demands is satisfied by `allowCustom` being a DECLARED axis on the node (the app declares the ACT), not by a per-value marker — that is the distinction to state in the record.

**OPEN-6 — MRU empty state: does the empty-query path reach the server at all? ⇒ It MUST, and the type must not preclude it.**
Reasoning: Salesforce's `searchType` defaults to `Recent`; the design only requires that we not PRECLUDE an app supplying `candidates` for an empty query. This plan's obligation is the wire half: **nothing in the type or its TSDoc may imply a minimum query length** (there is no `minChars` — D4, deliberately). **State affirmatively in the `searchAction` TSDoc that an empty query is a legitimate query and WILL be dispatched**, so an app can return most-recently-used candidates for it. The renderer half (the debounce must fire on a cleared input, not just a non-empty one) is enforced in Plan 21-03; cross-reference it here.

Feeds the popup listbox and **nothing else**. **NEVER the source of a selected label (D1).** 🚨 **ORDER IS MEANINGFUL APP DATA (D12). The renderer presents candidates AS GIVEN — it sorts nothing, dedupes nothing, and truncates nothing.** The app decides what comes back and in what order; relevance ordering is the SERVER's judgment, never the widget's. This is universal in mature pickers: Salesforce's picker `searchType` **defaults to `Recent`**, and Dynamics shows the five most-recently-used rows plus five favourites (explicitly NOT filtered by the search term). A renderer that "helpfully" alphabetized for tidiness would **silently destroy a server-side ranking with no way for the app to stop it** — a real, named consumer sorts candidates by recency-weighted mention frequency in their own provider handler. The server authorizes in the action handler with the real auth context, exactly as every other VMS action does; the picker only narrows what is *offered*. <threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| app-authored view tree → validators | An app (already-trusted per the VMS model) submits a ViewNode tree; the validators are a correctness gate, not a security gate |
| client → server (`bind` round-trip) | The ONLY authoritative lookup input. `selected`/`candidates` are server→client only and must never be trusted inbound (D1) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-21-01 | Tampering | collectActions field arm | mitigate | Record `searchAction` so a duplicate/ambiguous search action name is caught at build (`invalid_tree`), not silently dispatched to the wrong handler — the exact class of miss the real-validator shim rule (21-08) exists for |
| T-21-02 | Spoofing | `selected` / `candidates` treated as inbound | mitigate | The TSDoc states the direction invariant as a hard rule: server→client ONLY, never trusted coming back. The client physically sends only `bind`, so a forged label cannot reach a handler; enforced by shape, documented at the type |
| T-21-03 | Elevation of Privilege | picker `candidates` mistaken for an authorization boundary (D8) | mitigate | TSDoc states, in the `uploadOn` register, that the filter is UX and NEVER authorization — the server authorizes in the action handler with the real auth context. The wire carries no filter/ACL concept at all, so there is nothing to mistake for one at runtime |
| T-21-04 | Information disclosure | silent result truncation hiding an exact match (D7, the ServiceNow anti-pattern) | mitigate | No cap field on the wire; the TSDoc requires the app to render a VISIBLE `TextNode` ("Refine your filter — N matches, max is X"). Nothing truncates silently |
