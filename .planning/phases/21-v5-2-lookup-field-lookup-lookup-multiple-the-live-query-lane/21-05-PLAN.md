```typescript
// Unlike busy.test.ts's `stubFetch` (which auto-resolves in FIFO order),
// these tests need MANUAL, OUT-OF-ORDER control over exactly when each
// dispatch's response arrives — so the fetch mock here defers every
// post-load call until the test explicitly resolves it.
```
The harness (`makeControllableFetch`, :36-53) — the first call (the `load()` GET) resolves immediately; every subsequent call PARKS a deferred resolver the test resolves by hand:
```typescript
function makeControllableFetch(): { fetchMock: ReturnType<typeof vi.fn>; deferreds: Deferred[] } {
  const deferreds: Deferred[] = [];
  let callCount = 0;
  const fetchMock = vi.fn(() => {
    callCount++;
    if (callCount === 1) {
      return Promise.resolve(new Response(JSON.stringify({ vm: emptyVm, state: {}, ok: true }), { ... }));
    }
    return new Promise<Response>((resolve) => { deferreds.push({ resolve }); });
  });
  return { fetchMock, deferreds };
}
```
**This is what makes an interleaving scriptable:** fire dispatch A, fire dispatch B, then `resolveDeferred(deferreds[1], …)` BEFORE `deferreds[0]` — an out-of-order response, deterministically.

<decision_records>
Two questions are settled HERE and must be recorded in the SUMMARY with reasoning.

**OPEN-4 — stale-response discard-only vs cancel? ⇒ DECIDED: DISCARD-ONLY for v1. No `AbortController`, no cancellation.**
The design worried: "the superseded request still goes out and still costs the server. On a per-keystroke directory search that's real load." **The pattern-map showed this is better news than the design assumed, and the reasoning is worth recording precisely:** that concern is true only of requests ALREADY IN FLIGHT, and **two independent suppressors are stacked** before a request ever exists:
1. The **300ms debounce** (Plan 21-03) suppresses most keystrokes *before they reach `dispatch()` at all* — a fast typist produces roughly one dispatch per word, not per character.
2. The lane's **latest-wins coalescing slot** (index.ts:1305-1345) means rapid triggers never produce more than **2 concurrent requests total** — one in flight, one queued, overwrite-never-append.
⇒ The realistic worst case is ~2 requests per typing burst, not the "per-keystroke directory search" load OPEN-4 imagined. The `AbortController` argument is much weaker than it looks, cancellation is new machinery in the lane (which this phase is explicitly not touching — see 21-PATTERNS §3e), and discard-only is what react-select does and what we already have, built and tested. **Purely additive later if a real signal appears.** Record the decision WITH the two-suppressor reasoning, so a future phase reconsidering it starts from the arithmetic rather than the intuition.

**D4's open question — is a typeahead the signal for the deferred, conditional Phase 17 admission barrier? ⇒ DECIDED: NO. Not the signal. The barrier stays deferred.**
Reasoning: the admission barrier was recorded as conditional on a real signal of **lane contention** — many independent non-blocking producers fighting for one slot. A lookup is a **single producer**, already self-throttled by its own debounce, and the existing latest-wins coalesce slot is **precisely the right semantic for it**: an older query is worthless the instant a newer one exists, so it *wants* to be dropped, not queued. A barrier would add admission machinery to serve a producer that is happiest being superseded. **Record it so the question is CLOSED rather than re-asked next phase** — that is the whole point of writing it down.
