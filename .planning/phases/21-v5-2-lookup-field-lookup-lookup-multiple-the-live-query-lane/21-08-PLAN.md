<objective>
Document the picker as a public, first-class protocol in `agent-skill.md` (and its byte-identical .NET twin), and add the Showcase entry.

Purpose: 🏆 **this is the finding worth stating loudly.** *"There is essentially no prior art at the protocol level; everyone kept the picker's transport private."* ServiceNow's `/xmlhttp.do` is undocumented across all authoritative sources; Strapi's `/relations` endpoint has no doc page; Salesforce's UI API is the lone public exception. **In VMS the picker's search IS an ordinary action on the public wire** — an agent drives a lookup exactly like a human, with no special API, no private transport, and no undocumented endpoint. **We would be the first system where the reference picker is a documented, first-class protocol.** It costs us nothing; it falls straight out of the architecture. The design says explicitly: **this belongs in `agent-skill.md`.**

This is also a maintainer-rule obligation: the wire gained new field semantics, so the skill updates in the SAME change and is re-copied byte-identically. `parity/check-skill.ts` gates it — it diffs both source files AND the served HTTP bodies on the HelpDesk twins, so the .NET copy cannot silently drift.

Output: a lookup section in `viewmodel-shell/agent-skill.md`; a byte-identical `viewmodel-shell-dotnet/AgentSkill.md`; a Showcase entry.
</objective>

**The maintainer rule (AGENTS.md):** any change to the wire shape, response envelope, side-effect verb set, error code vocabulary, or polling semantics MUST update `viewmodel-shell/agent-skill.md` in the same change, then re-copy byte-identically to `viewmodel-shell-dotnet/AgentSkill.md` (embedded as the logical resource `AshleyShrok.ViewModelShell.AgentSkill.md`). `parity/check-skill.ts` diffs both source files AND the served HTTP bodies on the HelpDesk twins.

⚠️ **Judgement call to make and record.** Phase 20 concluded the skill needed NO change, on the reasoning that *"the skill enumerates the PROTOCOL, not the node catalog"* — breadcrumb/steps were new NODE types with no new protocol semantics. **The lookup is different and DOES warrant a section**, for two independent reasons:
1. It introduces **wire-field SEMANTICS an agent must know to drive it correctly** — specifically the D1 direction invariant (`selected` is display-only; a client-supplied label is meaningless and never trusted) and the fact that `bind` alone is authoritative. An agent that assumed `selected` round-trips would write labels into state and be silently wrong.
2. **Design §6 explicitly says so**, and it is the phase's stated differentiator (LOOK-07).
Contrast: toasts DID require a skill update (a new side-effect verb). Breadcrumbs did not (a pure node). The lookup sits with toasts, not breadcrumbs. **Record this reasoning in the SUMMARY** — the next node author will ask "does my node need a skill update?" and this is the discriminating test.

**The canonical source is `viewmodel-shell/agent-skill.md`**; the .NET file is a byte-identical copy. Copy with `cp`, never by hand — a hand-copy is how a byte diff happens.

**Design §6 — the content to convey, near-verbatim:**
> An agent sets `bind` to the id(s) and never touches the search UI. **It does not need to know the label** — `selected` is display-only and the server must never trust a client-supplied one (D1). The "an agent knows the id but not the label" wrinkle becomes an argument *for* the model.

**The Showcase (`demo/Showcase/`)** is the single source of truth for the design system's worked examples and carries a runtime 12-theme switcher in its gallery view. It imports the shipped stylesheet + themes; **the app writes ZERO CSS** (`<style>` blocks are forbidden; the only sanctioned deviation is a `:root{}` `--vms-*` token file imported after the theme).       - 🚨 **Candidate ORDER is meaningful and is preserved EXACTLY (D12).** The renderer presents `candidates` as given — it sorts nothing, dedupes nothing, truncates nothing. **For a wire-driving agent this cuts both ways:** when READING a lookup, the order IS the server's ranking (relevance / recency / MRU), so **position 0 is the server's best answer** — do not re-sort before choosing; when AUTHORING a provider handler, **the order you return is the order the user sees**, so ranking is yours to decide and yours to get right. Any cap is visible in the tree (D7), never silent.
      - `searchBind` + `searchAction`: the search **IS an ordinary action on the public wire** — an agent may POST `searchAction`'s name with the query in state at `searchBind`, read `candidates` off the returned tree, then set `bind` to the chosen id. **No special API, no private transport, no undocumented endpoint.**
      - An agent may skip the search entirely and set `bind` directly when it already knows the id — the common agent case.
      - `allowCustom` declares whether invented values are accepted; an invented value is just a value whose label equals itself (no bare strings, no union).
      - **An empty query is a legitimate query** and may return most-recently-used candidates (no minimum-length gate).
      - `blocking` is meaningless on `searchAction` (client-side lane selection; the renderer forces it, and it never rides the POST payload anyway).
      - **The picker's filter is UX, never authorization (D8)** — the server authorizes in the action handler; an agent must not infer an access boundary from what the picker offers.
      - **Any result cap is visible in the tree (D7)** — the app renders a `TextNode`; nothing truncates silently.
    - The section states the differentiator plainly: no surveyed platform publishes its picker's transport; here it is the same public wire everything else uses.
