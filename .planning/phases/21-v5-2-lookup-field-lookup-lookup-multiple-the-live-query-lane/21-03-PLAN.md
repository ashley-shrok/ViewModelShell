<decision_records>
Two of design §8's OPEN points are settled HERE because they are keyboard/rendering questions. Record each in the SUMMARY with its reasoning.

**OPEN-2 — does `Tab` select the active option or abandon it? ⇒ DECIDED: `Tab` CLOSES the popup and does NOT select. It abandons the active option; the field keeps whatever value it already had.**
⚠️ **APG is SILENT here** — its table only specifies where Tab *goes*, never what it does to the active option — so this is our call, and it will generate complaints either way. Reasoning: Tab is a NAVIGATION key, and a navigation key must never silently commit a value. The failure modes are asymmetric — Tab-abandons costs a user who wanted IDE/URL-bar accept semantics one extra keystroke (press Enter, then Tab), and they SEE that nothing was selected; Tab-accepts silently writes a wrong reference into a record when someone tabs past a field mid-typing, and **an accidental commit is UNANNOUNCED DATA CORRUPTION** — invisible to sighted users and doubly invisible to AT users. It also matches Escape's keep-the-value semantics (§7 item 18), so the two "get me out of here" keys behave consistently rather than one committing and one not. **Record this as a decision with this reasoning, and state it in the TSDoc**, because the next person WILL be asked "why doesn't Tab accept like my IDE?" and the answer must be findable.

**OPEN-5 — search error state: reuse `FieldNode.error` or add a distinct slot? ⇒ DECIDED: REUSE `FieldNode.error` for v1; record the `aria-invalid` overload as a known, documented wart.**
Reasoning: the differentiator is FREE and the principle is not negotiable — **react-select actively SWALLOWS fetch errors** (`loader.then(callback, () => callback())`), making a dead backend indistinguishable from "no results". No surveyed library has a first-class search-error state. That is a direct violation of principle 8 (nothing important fails quietly), and reusing `error` closes it at zero wire cost. The evidence (decorateField :1347-1357) says reuse gets us a `role="alert"` region + `aria-describedby` wiring for free, and design §7 item 9 reserves `assertive` for errors — so a genuine search FAILURE is a correct fit for that channel.
**The wart, documented not hidden:** `aria-invalid="true"` lands on the combobox, which is semantically wrong for "the server is down" (the user's input is not invalid), and it collides with a real validation error on the same field (one `error` slot, two meanings — last writer wins). Accept for v1: the app owns which message it puts there, the collision requires an app to have both a live search failure AND a pending validation error on the same field simultaneously, and a distinct slot is purely additive later. **Record the wart explicitly in the SUMMARY and in the `error` TSDoc's lookup note**, so a future phase can promote it to a distinct slot with the reasoning already written down. Do NOT silently swallow a search error under any circumstances — that is the one option this project has ruled out.

**OPEN-6 (settled in 21-01) — the renderer half is enforced in Plan 21-04** (the debounce must fire on a CLEARED input so an empty query reaches the server and an app can return most-recently-used candidates; no `minChars`, D4). D12 is the other half of OPEN-6 and IS enforced here: whatever the app returns for an empty query — an MRU list, ranked by its own judgment — is presented in exactly the order given.
