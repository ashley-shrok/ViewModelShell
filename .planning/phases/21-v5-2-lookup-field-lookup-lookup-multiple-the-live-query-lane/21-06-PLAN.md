<objective>
Ship `lookup-multiple`: the chips layer, its a11y baseline, and the `allowCustom` free-form path.

Purpose: D2 says single ships first, **multi immediately behind** — this is that. Multi is where the second-widget cost and **essentially all of the accessibility risk** live: the selected set itself becomes a second focusable dimension, and there is **no APG pattern for chips at all**. Everything here is extrapolation, so build conservatively.

⚠️ **The context that shapes this entire plan: the naive chips multi-select is a KNOWN, PUBLICLY FAILED design.** `alphagov/accessible-autocomplete-multiselect` carries the notice *"This project is retired as the component is not accessible."* It failed GOV.UK's own review because it *"does not announce the selections effectively or the presence of the 'Remove' button for screenreaders"*, and they judged the fixes *"will be challenging"* enough to **withdraw rather than repair it.** §7 items 25, 27 and 29 exist to prevent exactly that — they are not polish, they are the difference between this control and a retired one.

⚠️ **And `select-multiple` REMAINS the control for enumerable sets. That split is an ACCESSIBILITY REQUIREMENT, not taste** — the APG combobox has *"tested poorly with users for more than two decades"* (Roselli; Sarah Higley's testing lands similarly). **The lookup must NEVER try to swallow `select-multiple`.** Nothing in this plan touches the select branch.

Output: the multi path of the lookup arm (chips, roving tabindex, focus rule, two-step backspace, multi-listbox semantics, the `allowCustom` commit); `.vms-field__chip*` CSS; `test/lookup-multiple.test.ts`.
</objective>

**`.vms-badge` — the chip analog (default.css:1141-1185):**
```css
.vms-badge {
  --_badge-tone: var(--vms-text-muted);
  display: inline-flex;
  align-items: center;
  /* Hug content in ANY parent layout. A badge is semantically inline, but as a flex
     item it is BLOCKIFIED (inline-flex → flex) and a stack section's align-items:stretch
     would then stretch it to full width. Giving it a definite cross-size (fit-content)
     opts out of that stretch … */
  width: fit-content;
  vertical-align: middle;
  gap: var(--vms-space-2xs);
  font-size: var(--vms-text-xs);
  font-weight: 600;
  line-height: 1;
  padding: 0.2em 0.6em;
  border-radius: 999px;
  white-space: nowrap;
  /* default (no tone, no emphasis): subtle neutral tint pill. */
  background: color-mix(in srgb, var(--_badge-tone) 16%, var(--vms-surface));
  color: var(--_badge-tone);
  border: 1px solid transparent;
}
```
**The `--_badge-tone` private-var indirection is the technique to copy:** one private custom property holds the working color; the fill, text, and border all derive from it, so **every theme recolors automatically** because `--vms-error`/`--vms-surface` are theme-owned. `color-mix(in srgb, var(--_tone) 16%, var(--vms-surface))` is the **polarity-adaptive knockout** — mixing *toward `--vms-surface`* (not toward white/black) auto-adapts to light and dark with ZERO per-theme deepening. It is used 18× in this file; it is the house technique.

⚠️ **Two deliberate divergences from the badge:**
1. **Do NOT copy `white-space: nowrap`.** It conflicts with §7 item 33 — chip and option text must WRAP without truncation. The badge is a short status word; a chip holds a person's name. GOV.UK's 12× magnification tester: *"Some larger country names may take over two lines. It wasn't always clear to him that this had happened — the words just looked like separate suggestions that didn't make much sense."* ⇒ options must also be visually delimited so a wrapped option never reads as two.
2. **The chip is INTERACTIVE** (it contains a real remove `<button>`, §7 item 24); the badge is inert. It needs focus-visible styling on the remove button — `.vms-field__input:focus` (:537) is the token precedent.
**DO copy** `width: fit-content` and its comment's reasoning — the chip lives inside a flex chip-row and **will hit the identical blockification/stretch bug the badge comment (:1145-1152) documents.** That comment was written in 5.1.2 for exactly this failure mode; read it before writing the chip.

**BEM structure** (default.css:508-586): `.vms-field` block, `__element`, `--modifier`. `.vms-field--checkbox` (:578) is the precedent for an inputType-specific layout modifier.

**Roving tabindex** — VMS has one adjacent precedent (`TableRow.action` sets `tabindex=0` + `role="button"`), but **nothing in the codebase manages focus across a SET or restores focus after removing the focused element.** `<details>`/focus restore is by id, not by set position. This is one of only three items in the phase with no analog, and it is **Medium risk**: §7 item 29's next→previous→input rule is easy to get wrong and, per GOV.UK, is one of the two failures that retired their component. It gets its own jsdom test.

**SLDS precedent (D2):** Salesforce renders single-select INSIDE the input (no pill element exists at all) and multi as a **separate pill listbox OUTSIDE** the combobox with its own roving-tabindex focus model — two components in all but name.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| server `selected` labels → chip DOM + aria-label | Server strings become both visible text and accessible names |
| user chip edits → `bind` string[] | The authoritative multi-value; a wrong write is silent data corruption |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-21-19 | Tampering | chip label resolved from `candidates` (D1, multi path) | mitigate | The chip path reads `n.selected` only; the multi anti-trap test asserts chips keep their labels while `candidates` excludes them |
| T-21-20 | Information disclosure | XSS via a server-supplied chip label | mitigate | Labels set via `textContent`; `aria-label` is set via `setAttribute` (attribute value, never parsed as markup) — never `innerHTML` |
| T-21-21 | Repudiation | AT user cannot perceive the selection they are building (the GOV.UK retirement failure) | mitigate | §7 items 25/27/29 are implemented AND tested: unique item-specific remove names, add/remove announced WITH the running count, and the three-arm focus rule that never lands on `<body>` |
| T-21-22 | Tampering | single-press Backspace silently destroying a chip | mitigate | Two-step non-destructive backspace (§7 item 31): first press highlights + announces, second press removes — destructive-by-accident is designed out |
